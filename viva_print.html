📌 Project Overview (Say This First!)

ROOOMIE is a Roommate Compatibility Matching System built on MySQL. It lets hostel students register, fill lifestyle profiles, get auto-calculated compatibility scores, and send/accept pairing requests. It has a full audit trail, admin dashboard, and a Node.js REST API.

❓ Likely Viva Questions — Schema

Q: Why use ENUM instead of VARCHAR?

ENUM restricts values to a predefined set, saves storage (stored as integers internally), and prevents invalid data.

Q: What is ON DELETE CASCADE?

When the parent row is deleted, all child rows are automatically deleted. E.g., deleting a user auto-deletes their profile, pairings, sessions.

Q: What is ON DELETE SET NULL?

In activity_logs, if a user is deleted, their user_id becomes NULL instead of deleting the log. Preserves audit history.

Q: Why LEAST() and GREATEST() in the unique constraint?

Prevents duplicate pairings regardless of order. (1,5) and (5,1) both resolve to LEAST=1, GREATEST=5.

Q: Why utf8mb4 charset?

Supports full Unicode including emojis (4-byte). Regular utf8 in MySQL only supports 3-byte characters.

Q: What is a CHECK constraint?

Validates data before insert/update. E.g., CHECK (cleanliness_level BETWEEN 1 AND 5)

📁 FILE 2: 02_procedures.sql — Stored Procedures (7)

What is a Stored Procedure?

A precompiled set of SQL statements stored in the database. Supports IN/OUT parameters, variables, conditions, transactions. Called with CALL procedure_name(...).

Procedure 1: sp_create_user

CREATE PROCEDURE sp_create_user(
    IN  p_name VARCHAR(100), IN p_email VARCHAR(150),
    IN  p_password_hash VARCHAR(255), IN p_phone VARCHAR(20),
    IN  p_gender VARCHAR(10), IN p_bio TEXT,
    OUT p_user_id INT UNSIGNED   -- OUTPUT parameter
)

Steps:

  1. Validates name and email are not empty
  2. Checks duplicate email — IF EXISTS (SELECT 1 FROM users WHERE email = ...)
  3. Starts TRANSACTION — ensures atomicity
  4. Inserts into users with TRIM(), LOWER(), NULLIF()
  5. Gets new ID — SET p_user_id = LAST_INSERT_ID()
  6. Auto-creates empty profile row
  7. Logs the event to activity_logs with JSON
  8. COMMIT — if anything fails, ROLLBACK via EXIT HANDLER

Key concepts: SIGNAL SQLSTATE '45000' (custom error), DECLARE EXIT HANDLER (catches errors), NULLIF(x,'') (empty→NULL), LAST_INSERT_ID()

Procedure 2: sp_update_user

Snapshots old values before updating for audit. Uses IFNULL(NULLIF(TRIM(p_name),''), name) — keeps old value if new is empty.

Procedure 3: sp_delete_user

Supports soft delete (status='inactive') and hard delete (DELETE FROM users). Both logged with admin ID.

Procedure 4: sp_save_profile

Uses INSERT ... ON DUPLICATE KEY UPDATE (UPSERT). If profile exists → update; if not → insert.

Procedure 5: sp_send_pairing_request (Most Complex)

  1. Self-pairing guard
  2. Both users must be active
  3. No existing active pairing (checks with LEAST/GREATEST)
  4. Calls fn_compatibility_score() to auto-calculate
  5. Inserts pairing with 7-day expiry
  6. Creates notification for recipient
  7. Logs the event

Procedure 6: sp_respond_to_pairing

Uses SELECT ... FOR UPDATE (row-level locking). Only recipient can respond. Only pending requests accepted. Notifies sender.

Procedure 7: sp_cancel_pairing

Only the initiator can cancel. Only pending requests can be cancelled.

❓ Likely Viva Questions — Procedures

Q: Difference between IN and OUT parameters?

IN = input (read-only). OUT = output returned (write-only). INOUT = both.

Q: What is a TRANSACTION?

Groups SQL statements into one atomic unit. ALL succeed (COMMIT) or ALL fail (ROLLBACK). In sp_create_user, if profile insert fails, user insert is rolled back.

Q: What is SIGNAL SQLSTATE?

MySQL's way to throw custom errors. '45000' is user-defined range. We use '45001' for duplicate email, '45003' for self-pairing.

Q: What is ON DUPLICATE KEY UPDATE?

If INSERT violates UNIQUE/PRIMARY key, updates existing row instead. It's an UPSERT (Insert or Update).

Q: What is SELECT ... FOR UPDATE?

Locks selected rows so no other transaction can modify them. Prevents race conditions.

Triggers (4)

Trigger 1: trg_after_user_insert (AFTER INSERT ON users)

Auto-logs creation event with JSON of name, email, gender, role. Uses NEW.user_id, NEW.name.

Trigger 2: trg_after_user_update (AFTER UPDATE ON users)

Detects which fields changed by comparing OLD vs NEW. Builds JSON diff with JSON_SET().

Trigger 3: trg_before_pairing_insert (BEFORE INSERT ON pairings)

  1. Normalizes IDs — smaller ID always stored as user_id_1
  2. Sets default expiry — 7 days via DATE_ADD(NOW(), INTERVAL 7 DAY)
  3. Auto-calculates score — calls fn_compatibility_score()

Trigger 4: trg_after_pairing_update (AFTER UPDATE ON pairings)

Logs status transitions only when status actually changed.

Event: evt_expire_pairings

CREATE EVENT evt_expire_pairings
ON SCHEDULE EVERY 1 HOUR
DO BEGIN
    UPDATE pairings SET pairing_status = 'expired'
    WHERE pairing_status = 'pending' AND expires_at < NOW();
END

Requires SET GLOBAL event_scheduler = ON

❓ Likely Questions — Triggers

Q: BEFORE vs AFTER trigger?

BEFORE fires before row is written — can MODIFY NEW values. AFTER fires after — can only READ NEW/OLD.

Q: What are NEW and OLD?

NEW = row being inserted/updated. OLD = row before update/delete. INSERT has only NEW, DELETE has only OLD, UPDATE has both.

Q: Why normalize IDs in BEFORE INSERT?

So (5,1) and (1,5) are stored as (1,5). Makes UNIQUE constraint work correctly.

❓ Likely Questions — Views

Q: View vs Table?

View = virtual, no data stored, always latest. Table = physical, data on disk.

Q: Can you INSERT through a view?

Simple views (single table) — yes. Complex views (joins, GROUP BY) — no.

Q: What is LEFT JOIN?

Returns ALL rows from left table even if no match in right (NULL for unmatched).

📁 Backend: server.js — Node.js REST API

Tech: Express.js + mysql2 + bcryptjs + cors

  • Connection pooling: mysql.createPool() — reuses connections
  • Password hashing: bcrypt.hash(password, 12)
  • SQL injection prevention: parameterized queries with ? placeholders
  • Error handling: asyncHandler wraps routes; SQLSTATE mapped to HTTP codes

Q: What is connection pooling?

Maintains reusable DB connections. Connections borrowed and returned. connectionLimit: 10.

Q: How is SQL injection prevented?

Parameterized queries — pool.execute('SELECT * FROM users WHERE id = ?', [userId])

Q: Why bcrypt?

One-way, salted, slow hashing. Even if DB is compromised, passwords can't be reversed.
